# ADR 0010：ACP 权限由浏览器端显式审批

## 状态

已被 ADR 0026 取代，2026-07-15。V0.1 的一次性任务没有浏览器权限审批。

## 背景

本地 Claude 由 `agentferryd` 通过 ACP 持有会话。Agent 可能在扩展弹窗关闭后请求修改文件、执行命令或进行其他需要授权的操作，因此权限请求不能依赖一个持续打开的弹窗，也不能因为用户点击了“交接”就自动获得无限授权。

## 决策

V0.1 采用以下权限审批流程：

1. daemon 收到 ACP 权限请求后，将任务切换为 `awaiting_approval`；
2. Chrome 扩展通过通知和扩展角标提醒用户；
3. 用户重新打开扩展，查看操作类型、目标、Agent 提供的理由和风险提示；
4. 用户明确批准或拒绝后，daemon 将结果返回 ACP Session；
5. 未处理的权限请求保持暂停，超过配置的时限后默认拒绝。

点击“交接”只授权创建会话并发送首条消息，不代表预先批准后续敏感操作。V0.1 不提供“永久允许全部操作”。具体超时时长和是否允许对低风险操作配置有限的自动批准规则，继续作为产品配置细节讨论。

## 后果

- daemon 必须持久保存任务与待审批状态，不能依赖扩展弹窗生命周期；
- 扩展需要权限请求列表、详情、批准和拒绝操作；
- Chrome 通知不可用时，扩展角标和任务列表仍需保留待审批状态；
- 通知内容不得暴露捕获正文、凭据或其他敏感参数；
- daemon 重启后必须安全恢复或拒绝悬空的权限请求，不能默认批准。
